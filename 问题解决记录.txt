问题解决记录

一、不调用工具直接进行回答

问题现象：
在测试中输入“计算 2+2”时，模型直接返回了 final_answer，没有调用 calculator 工具。虽然结果正确，但不符合希望演示的 Agent 工具调用链路。

问题原因：真实 LLM 本身具备一定计算能力，面对简单表达式时，会倾向于直接回答，而不是主动调用工具。这说明仅靠模型自由判断，无法稳定触发工具调用。对于 Agent Runtime 来说，工具是否调用不能完全依赖模型习惯，而需要通过工具选择规则进行约束。

问题解决：要在 system prompt 里加硬规则，只要用户请求数学计算，就必须调用 calculator 工具，不允许模型直接心算。

二、跨轮追问失败，模型返回格式无法解析

问题现象：
在读取文档并总结后，继续追问“刚才那个任务执行成功了吗？调用了什么工具？”时，程序返回“模型返回了无法识别的格式”。也就是说，模型没有按照约定的 JSON 协议输出，runtime 无法解析。

问题原因：在读取文档并总结后，模型没有按照约定的 JSON 协议输出，runtime 无法解析；session_state 只保存 last_* 字段，新的 run 会覆盖旧的 run 信息，导致跨轮追问时无法稳定追溯最近几轮任务状态。

问题解决：一方面，增强 JSON 解析能力：runtime 支持解析标准 JSON、去除 markdown 代码块、从混合文本中提取 JSON 对象；如果仍解析失败，则将 raw_response 写入 trace，方便定位。另一方面，调整 session_state 的结构，不再只保存单一 last_* 字段，而是拆分为多个部分，既能记录最近一次输入，也能保留最近成功任务和最近若干轮执行摘要。

三、通过 trace 发现状态查询不适合完全交给 LLM

问题现象：
在输入“最近几轮我让你做了什么？”时，trace 中显示模型其实已经根据历史记录生成了正确的自然语言回答，但由于没有包装成 JSON，runtime 仍然判定为解析失败。

问题原因：
状态查询类问题不一定适合继续走 LLM，像“最近几轮做了什么”“上一次失败原因是什么”“刚才调用了什么工具”这类问题，本质是查询 session_state，不需要模型重新推理，也不需要调用工具。

解决思路：在 runtime 中增加轻量状态查询识别逻辑。对于明显的 session 状态查询，例如“最近几轮我让你做了什么”“上一次失败的任务是什么”，runtime 直接读取 session_state 生成回答，不再调用 LLM。
与此同时，给 run 增加 run_type 字段，区分 task 和 state_query。状态查询可以写入 recent_runs，但不能覆盖 last_task_run 和 last_successful_task_run，避免“查询状态的问题”反过来覆盖真正的任务状态。


四、重复调用
问题现象：
在测试“读取 agent_runtime.md，并总结里面的核心内容”时，read_docs 工具被连续调用了两次，直到第三步模型才返回 final_answer。

问题原因：
LLM 在拿到第一次 observation 后，没有意识到工具已经执行成功，仍然再次发起相同的 tool_call。这说明仅靠 prompt 提醒“不重复调用”不够稳定，runtime 需要具备重复工具调用保护。

解决思路：
在 runtime 中维护当前 run 的 seen_tool_calls。每次工具调用前，根据 tool_name 和 arguments 生成唯一 signature。如果同一个 signature 在当前 run 中已经成功执行过，则阻止重复执行，并记录轨迹。此时 runtime 可以复用已有 observation，提示模型基于已有 observation 生成 final_answer。这样即使模型重复请求同一个工具，程序也不会真的重复执行。
